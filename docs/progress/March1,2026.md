### 2.2 Federation Configurations (addresses issue #1)

Expanded from 4 configs to 6, including specialty-based distribution:

| Config | Structure | Description |
|--------|-----------|-------------|
| `centralized_1do` | 1 DO × 100% | Baseline — all data centralized |
| `separated_3do` | 3 DOs × 1 dataset each | DO1=StatPearls, DO2=Textbooks, DO3=MIMIC |
| `hybrid_2do` | 2 DOs × 50% all | Each DO holds 50% of all 3 datasets |
| `hybrid_3do` | 3 DOs × 33% all | Each DO holds 33% of all 3 datasets |
| `hybrid_4do` | 4 DOs × 25% all | Each DO holds 25% of all 3 datasets |
| **`specialty_7do`** (new) | 7 DOs by specialty | Each DO holds specialty-specific docs (cardiology, oncology, neurology, etc.) |

### 2.3 Merging Strategies (addresses issue #7)

Expanded from RRF-only to 5 strategies:

| Strategy | Formula | Status |
|----------|---------|--------|
| RRF (baseline) | `score(d) = Σ 1/(k + rank(d))` | Evaluated on all models × all configs |
| Borda Count | `score(d) = Σ (n - rank(d))` | Evaluated on OpenBioLLM-70B × specialty_7do + hybrid_3do |
| CombSUM | `score(d) = Σ normalized_score(d)` | Evaluated on OpenBioLLM-70B × specialty_7do + hybrid_3do |
| CombMNZ | `score(d) = |sources(d)| × Σ normalized_score(d)` | Evaluated on OpenBioLLM-70B × hybrid_3do |
| Weighted RRF | `score(d) = Σ w_i/(k + rank(d))` | Evaluated on OpenBioLLM-70B × specialty_7do + hybrid_3do |

### 3.4 Key Findings from Current Results

1. **Model scale dominates accuracy**: 70B models (73-79%) >> 7B models (47-52%) >> 1.7B (39-42%)
2. **Bio-specialization helps at 70B but not 7B**: OpenBioLLM-70B outperforms Llama-3.3-70B by 3-9pp across configs. BioMistral-7B does NOT outperform Mistral-7B, suggesting specialization benefits scale with model capacity.
3. **Federation can beat centralized**: hybrid_2do achieves 78.76% for OpenBioLLM-70B vs 75.33% centralized (+3.4pp). Moderate federation improves retrieval diversity.
4. **Over-fragmentation hurts**: specialty_7do yields the lowest accuracy for most models due to knowledge fragmentation across 7 specialized DOs.
5. **Merger differences are small but real**: ~1.6pp range in specialty_7do (where merging matters most). Weighted RRF is consistently best. In hybrid_3do with uniform data, merging strategy is negligible.
6. **PubMedQA is the hardest dataset** (30-65% across models), BioASQ the easiest (58-83%).

## 4. Remaining Reviewer Issues & Plan to Address

### Issue #2: Approval Mechanism (High Priority)

**Status**: Not yet implemented.

**Plan**:
- Implement `approval_rate` parameter in federation config
- Test 4 approval rates: 100% (baseline), 75%, 50%, 25%
- Two rejection modes:
  - **Random rejection**: each DO independently rejects each query with probability (1 - approval_rate)
  - **Selective rejection**: specific DOs always reject (simulates a hospital opting out entirely)
- Frame results as **privacy-utility tradeoff**: lower approval = more privacy but less accuracy
- Key research question: how gracefully does accuracy degrade? Which DOs are most critical?

**Estimated effort**: Low — primarily config changes + re-running existing pipeline with DO subsets.

### Issue #3: Privacy / Adversarial Evaluation (High Priority)

**Status**: Not yet addressed.

**Plan (pragmatic approach given resource constraints)**:

#### 3a. Information Leakage Analysis (no extra runs needed)
- From existing results, calculate: how many unique corpus chunks are exposed to the DS through top-k retrieval across all 500 queries?
- Measure **corpus exposure rate** = (unique chunks returned) / (total chunks in DO corpus) per DO
- Show this metric across federation configs — more DOs = each DO exposes less, but total exposure may increase

#### 3b. Reframe Privacy Claims
- **Reduce claims**: position the paper as a federated RAG *performance* benchmark, not a privacy-preserving system
- Describe privacy as an **architectural property** (data stays local, only snippets move) rather than a formal guarantee
- Explicitly acknowledge that top-k snippets leak relevant data to the DS

#### 3c. Discuss (but don't necessarily implement) Threat Scenarios
- **Membership inference**: Can a malicious DS determine if a specific document exists in a DO's corpus?
- **Corpus reconstruction**: After N queries, what fraction of a DO's corpus can DS reconstruct?
- **Query intent inference**: Can a compromised DO infer the DS's research intent from query patterns?
- Present these as important future work with concrete formulations rather than vague hand-waving

#### 3d. Optional: Basic Privacy Mechanism
- If time permits, add differential privacy noise to retrieval scores as a simple privacy mechanism and measure accuracy-privacy tradeoff
- This would significantly strengthen the paper but is not strictly required if claims are properly scoped

### Issue #4: Statistical Rigor (Medium Priority)

**Status**: Not yet addressed.

**Plan**:
- **Primary defense**: If using temperature=0 / greedy decoding, results are **deterministic** — the same input produces the same output. Replicates would be identical, making standard errors inapplicable. State this explicitly.
- **Bootstrap confidence intervals**: Resample the 500 per-question binary outcomes (correct/incorrect) to compute 95% CIs. This is computationally free and gives error bars.
- **Cross-dataset variance**: Report mean ± std across the 5 QA datasets as a robustness measure.
- **Acknowledge hardware constraints**: State total experiment time and hardware (MacOS, local machine) transparently.

### Issue #5: Interpretability / Theoretical Grounding (Medium Priority)

**Status**: Partially addressed by merger strategy comparison.

**Plan**:
- **Analyze retrieval quality separately from generation quality**: measure Recall@k and MRR for retrieval across configs, independent of LLM accuracy. This separates "did federation hurt retrieval?" from "did the LLM fail to use the context?"
- **Explain the hybrid_2do > centralized finding**: hypothesis — moderate federation creates retrieval diversity (different index partitions surface different relevant documents). Test by measuring document overlap across DOs in returned results.
- **Explain specialty_7do degradation**: hypothesis — specialty fragmentation means most queries only get relevant results from 1-2 of 7 DOs, diluting the top-k with irrelevant snippets from the other 5. Test by measuring per-DO retrieval relevance scores.
- **Explain bio-specialization × scale interaction**: at 7B scale, the model isn't capable enough to benefit from better retrieval context. Show this by comparing accuracy with vs. without RAG (no-retrieval baseline).

### Issue #6: Missing Literature (Medium Priority)

**Status**: Not yet addressed.

**Plan**:
- Add citations to distributed information retrieval literature (Callan 2000, Si & Callan 2003)
- Add citations to secure ranking aggregation
- Cite private kNN and private top-k retrieval work
- Cite secure multi-party computation literature relevant to federated retrieval
- Position the paper within the federated IR landscape, not just the FL landscape

### Issue #8: Experimental Dimensions (Medium Priority)

**Status**: Partially addressed (more QA datasets, more configs, more mergers).

**Still missing**:

- Query routing (don't send query to all DOs) — interesting but complex

**Plan**: Prioritize **top-k sensitivity analysis** (low effort, directly addresses Reviewer C's question). Consider one alternative embedding model if time permits.
